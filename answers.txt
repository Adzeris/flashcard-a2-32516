1. WHAT ARE THE ENTITIES AND WHERE'S THE CRUD?
================================================================

Five SQLAlchemy models in backend/app/models.py:

- User           (id, username, email, hashed_password, role, created_at)
- Test           (id, user_id FK, name, created_at)            <- per-user folder
- Flashcard      (id, user_id FK, question, answer, category, difficulty, ...)
- ViewHistory    (id, user_id FK, flashcard_id FK, notes, was_correct, viewed_at)
- GuestSession   (id, token, title, flashcards_json, created_at, expires_at)

Required by rubric: 3 entities with CRUD. I have at least 4 with CRUD.

CRUD per entity:
- Users:        register/POST, /me GET, /me PUT, admin GET/PUT/DELETE on /api/users
- Tests:        GET/POST/PUT/DELETE /api/tests
- Flashcards:   GET/POST/PUT/DELETE /api/flashcards (?search & ?test for filtering)
- ViewHistory:  GET/POST/PUT/DELETE /api/histories
- GuestSession: POST /api/guest/sessions, GET/PUT/DELETE /{token}

2. WHY useState AND NOT useReducer / useRef / useContext?
================================================================

useState:
- Most state in this app is independent (a form value, a tab name, a list).
  No "many actions, one shape" kind of transitions, so useReducer would just
  be ceremony.

useReducer would make sense if I had something like a wizard with many step
transitions, or a complicated undoable list - I don't.

useRef:
- I don't need a mutable value that survives renders without triggering one,
  and I don't need to reach into a DOM node imperatively. So no useRef.

useContext:
- The auth state IS shared across every screen. I deliberately did NOT use
  context for it because I only have one consumer level deep (App passes
  token down to each tab). Context would just hide the data flow.
- If the tree got deeper, e.g. "Layout > Tabs > SubTab > NestedForm needs
  the token", I'd promote useAuth into a context provider. Not worth it now.

Custom hooks (useAuth, useUiFeedback):
- useAuth packages token + user + login/logout/refresh. Without it, App.jsx
  would have to handle localStorage, /me fetching, and 401 cleanup itself.
- useUiFeedback wraps the loading/error/success try/catch pattern that every
  view needed. Saves about 8 lines per handler and means error messages
  always appear in the same place.

3. HOW DOES JWT WORK HERE?
================================================================

Login flow (backend/app/routers/auth_router.py + auth.py):

1. Frontend POSTs username/password as form data to /api/auth/login.
2. Backend looks up user, runs passlib.verify_password(plain, hashed).
   We use PBKDF2-SHA256 (set in auth.py CryptContext) - chose it over
   bcrypt because bcrypt has a 72-byte input cap which was annoying.
3. If valid, jose.jwt.encode signs a token with HS256 using SECRET_KEY
   from env (falls back to a dev string only if the env isn't set).
   Payload is just {"sub": str(user_id), "exp": <utc timestamp>}.
4. Token is returned and the frontend stores it in localStorage.

Authenticated request flow:

1. Frontend's authRequest in api.js sends "Authorization: Bearer <token>".
2. Backend dependency get_current_user (deps.py) reads the bearer token
   via OAuth2PasswordBearer, decodes with the same SECRET_KEY/ALGORITHM,
   pulls the user_id out of "sub", looks the user up, raises 401 on any
   failure.
3. Routes that need auth declare:  current_user = Depends(get_current_user)

Token expiry: ACCESS_TOKEN_EXPIRE_MINUTES (default 180 min = 3 hours).

Why a Bearer header and not a cookie?
- No CSRF concern because the browser doesn't auto-attach Authorization
  headers to cross-site requests like it does with cookies.
- localStorage has its own risks (XSS), but I have no third-party scripts
  and React escapes everything by default, so the trade is fine.

4. HOW IS ROLE-BASED ACCESS CONTROL ENFORCED?
================================================================

Backend is the source of truth. deps.py defines:

    def get_admin_user(current_user = Depends(get_current_user)):
        if current_user.role != "admin":
            raise HTTPException(403, "Admin access required")
        return current_user

Admin-only routes (e.g. listing all users, deleting a user) declare:

    admin_user = Depends(get_admin_user)

Frontend just hides the admin tab when user.role != "admin" (App.jsx
filters availableTabs). If someone manually switched the tab or hit the
endpoint with curl using a non-admin token, the backend would still return
403. UI hiding is a UX nicety, not a security control.

Object-level checks: in flashcards_router and histories_router, regular
users can only modify rows they own (card.user_id != current_user.id -> 403).
Admins skip the ownership check. This is in addition to role checks.

5. HOW DOES LIVE SEARCH WORK?
================================================================

In components/TestWorkspace.jsx:

    useEffect(() => {
      const timer = setTimeout(() => {
        // GET /api/flashcards?test=<name>&search=<term>
      }, 200);
      return () => clearTimeout(timer);
    }, [search, test, token, ui]);

- Each keystroke updates the `search` state, which retriggers the effect.
- The effect schedules a fetch in 200ms.
- The cleanup cancels the previous timeout, so we only actually fire if
  the user pauses typing for 200ms. This is debouncing.
- Backend filters with ILIKE on question OR answer (flashcards_router.py).

Without the debounce we'd hit the API on every keystroke - fine for SQLite
locally, bad practice for any real backend.

6. WHY SQLITE? WHY NOT MONGO/POSTGRES?
================================================================

- SQLite is one file, no server to install, the database creates itself
  when the backend starts. Good for marker / new machine.
- The data is fully relational (users own tests, tests group flashcards,
  histories link users to flashcards). A relational DB is the natural fit.
- If this were a real product I'd use Postgres. The code wouldn't change
  much because SQLAlchemy abstracts it - just swap the connection string
  in database.py.

I deliberately did NOT pick MongoDB because forcing relations into a
document store for relational data is a bad fit, and it would require
installing MongoDB just to mark the assignment.

7. WHY FASTAPI?
================================================================

- Type hints + Pydantic give me request/response validation for free
  (return wrong shape -> 500 in dev, fix it).
- Auto OpenAPI docs at /docs help with manual testing.
- Dependency injection via Depends made get_current_user / get_admin_user
  reusable across every router without inheritance.
- Async-ready, though I'm using sync SQLAlchemy here for simplicity.

8. SPA WITHOUT REACT ROUTER - IS THAT REALLY AN SPA?
================================================================

Yes. The rubric just says "behave like an SPA" with one .html file and
no full-page reloads. I do that by:

- Keeping `tab` as a string in App.jsx state.
- Conditionally rendering one of FlashcardsTab/HistoryTab/ProfileTab/AdminTab.
- Mounting modal screens (study session) via createPortal to document.body.
- Using window.history.replaceState for the guest URL parameter so the
  share link survives reload without ever triggering a navigation.

I didn't add React Router because there's no need for back-button history
between tabs and no deep-linkable views (besides the guest token, which
I handle with a URLSearchParams read on mount). Adding it would have
been complexity for no win.

9. WHAT'S PROTECTED VS PUBLIC?
================================================================

Public (no token needed):
- POST /api/auth/register
- POST /api/auth/login
- All /api/guest/sessions endpoints (anyone with the token = anyone with
  the URL can edit; expiry kills access).
- GET /health

Authenticated (any logged-in user):
- /api/auth/me
- /api/tests, /api/flashcards, /api/histories (own data only)
- /api/users/me

Admin only:
- GET /api/users
- PUT/DELETE /api/users/{id}
- GET /api/users/{id}/history

10. ERROR HANDLING - WHAT HAPPENS IF THE BACKEND IS DOWN?
================================================================

- api.js wraps every fetch and parses { detail } from FastAPI errors into
  a readable Error message. Validation arrays also get unwrapped to the
  first message instead of dumping a JSON blob.
- In App.jsx and the tab components, the useUiFeedback hook catches errors
  and surfaces them in the .error <p> at the top of the screen.
- The user never sees a blank white page from a thrown promise. The worst
  case is "Failed to fetch" or a backend 500 message in red text.

12. WHY DID YOU SPLIT App.jsx LIKE THAT?
================================================================

Originally App.jsx was ~1300 lines after assignment 1 features got tacked
on. That was painful to work in: every state change rerendered everything,
Cmd+F was the navigation, and any teammate would have hated me.

The split principle: one file per "screen" (tab) the user actually sees,
plus shared widgets (DifficultyPicker, StudyPanel) and shared state
(useAuth, useUiFeedback) extracted into components/ and hooks/. App.jsx
is now just an orchestrator - read it top to bottom and it tells you
what view shows when. That made the design rationale defensible too.
